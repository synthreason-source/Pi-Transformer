A computer agent is a software system that perceives its environment and takes actions to achieve a goal. An agent can operate autonomously without constant human supervision. Agents differ from simple scripts because they make decisions based on changing conditions. A reactive agent responds directly to inputs without maintaining internal state. A deliberative agent plans several steps ahead before acting. Many modern agents are built on top of large language models. A language model agent breaks a task into smaller subtasks. The agent selects a tool, calls it, and observes the result. Observation of the result lets the agent decide the next action. This loop of observing, deciding, and acting is often called the agent loop. Some agents use a scratchpad to record intermediate reasoning. A scratchpad helps the agent keep track of partial progress. Tool use lets an agent extend its capabilities beyond text generation. Common tools include web search, code execution, and file access. A web browsing agent can navigate pages and click links. A coding agent can read a repository and propose changes. A coding agent can also run tests to check its own work. An agent with file access can create, edit, and delete files. Some agents can call other agents to delegate subtasks. Delegation allows a single request to be split among specialized agents. A planner agent decides the overall strategy for a task. A worker agent executes a narrow, well defined step. A critic agent reviews the output of another agent for errors. A memory system lets an agent recall facts from earlier in a session. Long term memory allows an agent to remember information across sessions. Short term memory holds only the current conversation context. Context windows limit how much text an agent can consider at once. When context grows too large, an agent may need to summarize it. Summarization trades detail for the ability to keep going. An agent framework provides reusable components for building agents. Popular frameworks handle tool calling, memory, and orchestration. Orchestration coordinates multiple agents or multiple steps. A pipeline is a fixed sequence of steps executed in order. An agent is more flexible than a pipeline because it can change its plan. Agents can fail when a tool returns an unexpected result. Good agents include error handling for failed tool calls. Retrying a failed action is a simple form of error recovery. Some agents ask a human for help when they are uncertain. Human in the loop design keeps a person able to intervene. A fully autonomous agent acts without asking for permission. Autonomy increases risk if the agent misunderstands its goal. Alignment work tries to make an agent's goals match human intent. A reward signal tells an agent whether its actions were good. Reinforcement learning agents improve by trial and error. Supervised agents are trained on examples of correct behavior. An agent's policy is the rule that maps observations to actions. A value function estimates how good a state is for the agent. Exploration lets an agent try new actions to learn more. Exploitation lets an agent use what it already knows works. Balancing exploration and exploitation is a classic agent problem. A multi agent system contains several agents interacting together. Multi agent systems can cooperate toward a shared goal. Multi agent systems can also compete against each other. Communication protocols let agents exchange messages reliably. A message bus can route messages between many agents. An agent's action space defines everything it is allowed to do. Restricting the action space can make an agent safer. Sandboxing runs an agent's code in an isolated environment. Sandboxing limits the damage a buggy agent script can cause. Logging records what an agent did and why. Logs help developers debug an agent's unexpected behavior. An agent's prompt describes its role, goals, and constraints. A system prompt is set once and applies to the whole session. A user prompt changes with each new request. Few shot examples in a prompt show the agent how to behave. Chain of thought prompting encourages an agent to reason step by step. Self reflection lets an agent critique its own previous output. An agent that revises its plan after failure is more robust. Task decomposition breaks a large goal into smaller pieces. Each subtask can be assigned to the most suitable agent. A router agent decides which specialized agent should handle a request. A retrieval agent searches a knowledge base for relevant documents. Retrieval augmented generation combines search with text generation. Grounding an agent's answers in retrieved documents reduces errors. Hallucination happens when an agent states something false with confidence. Verification steps can catch a hallucination before it reaches the user. An evaluation suite measures how well an agent performs on test tasks. Benchmarks compare different agents on the same set of tasks. Latency measures how long an agent takes to respond. Cost measures how many resources an agent consumes per task. A cache can store previous results so an agent avoids repeated work. Rate limits control how often an agent may call an external service. An API key authenticates an agent when it calls a service. Permissions define what actions an authenticated agent may take. An audit trail records every action an agent has performed. Guardrails constrain an agent to safe and expected behavior. A policy filter can block an agent from taking a disallowed action. Testing an agent in simulation avoids risk in the real world. A digital twin mirrors a real system for safe agent testing. Deployment moves an agent from testing into real world use. Monitoring watches a deployed agent for signs of failure. Alerts notify a human when an agent behaves unexpectedly. Rollback reverts an agent's changes if something goes wrong. Version control tracks changes to an agent's code and prompts. A changelog documents what changed between agent versions. Continuous integration tests an agent automatically after each update. A feedback loop lets user corrections improve future agent behavior. Fine tuning adjusts a model's weights using new training data. Prompt engineering adjusts an agent's instructions without retraining. Few shot learning lets an agent adapt using only a handful of examples. Zero shot performance measures an agent with no task specific examples. An agent's context includes the conversation history and any retrieved data. Trimming context helps an agent stay within its token budget. Token budgets limit how much text a model can process at once. Streaming lets an agent's output appear as it is generated. Asynchronous agents can work on multiple tasks at the same time. A queue holds tasks waiting for an available agent. A scheduler decides when each queued task should run. Idempotent actions can be safely retried without side effects. A dry run lets an agent preview an action before committing to it. Confirmation steps ask a user to approve risky agent actions. Explainability helps a user understand why an agent chose an action. Interpretability tools inspect an agent's internal reasoning. Transparency builds trust between a user and an autonomous agent. An agent's goal should be specified clearly to avoid ambiguity. Ambiguous goals can lead an agent to unintended behavior. Specification gaps let an agent satisfy the letter but not the spirit of a goal. Reward hacking occurs when an agent exploits a flawed reward signal. Robustness testing checks how an agent handles unexpected inputs. Adversarial testing tries to make an agent fail on purpose. Red teaming probes an agent for security or safety weaknesses. Access control limits which data an agent is allowed to read. Data minimization limits how much personal data an agent collects. Privacy preserving agents avoid exposing sensitive user information. An agent's environment can be physical, like a robot, or virtual, like software. A robotic agent perceives the world through sensors. A robotic agent acts on the world through actuators. A software agent perceives the world through APIs and files. A software agent acts on the world by calling functions. Simulation environments let researchers train agents cheaply and safely. Curriculum learning gradually increases task difficulty for an agent. Transfer learning lets an agent reuse skills learned on one task for another. Generalization measures how well an agent performs on unseen tasks. Overfitting happens when an agent memorizes training tasks instead of learning general skills. Regularization techniques help prevent an agent from overfitting. An ensemble combines multiple agents to improve overall performance. Voting lets an ensemble of agents agree on a final answer. A consensus mechanism resolves disagreement between multiple agents. Trust scores can weight the influence of different agents in a system.

An ontology defines a set of concepts and the relationships between them. A concept in an ontology is often called a class or a category. An instance is a specific example that belongs to a class. A relationship links two concepts and describes how they connect. The is a relationship links a specific class to a more general class. The part of relationship links a component to the whole it belongs to. The has a relationship links a class to a property or attribute it possesses. A taxonomy is a hierarchy of classes connected by is a relationships. A mereology is a system for describing part of relationships. An upper ontology defines very general concepts shared across domains. Examples of upper ontologies include SUMO, BFO, and DOLCE. A domain ontology defines concepts specific to one field of knowledge. A medical ontology like SNOMED CT models diseases, symptoms, and treatments. A biological ontology like the Gene Ontology models genes and their functions. The Gene Ontology links genes to biological processes they participate in. The Gene Ontology also links genes to cellular components where they act. A geographic ontology models places, boundaries, and spatial relationships. The contains relationship links a larger region to the smaller regions inside it. The adjacent to relationship links two regions that share a border. A library classification system is a kind of subject ontology. The Dewey Decimal System groups books into numbered subject categories. Library of Congress Classification groups books using letter and number codes. A linguistic ontology like WordNet links words by meaning rather than spelling. WordNet groups words with similar meanings into sets called synsets. The synonym relationship links words that share the same synset. The hypernym relationship links a specific word to a more general word. The hyponym relationship links a general word to a more specific word. The antonym relationship links two words with opposite meanings. A product ontology in e commerce links products to categories and attributes. The compatible with relationship links products that work together. The requires relationship links a product to another product it depends on. An organizational ontology models roles, departments, and reporting lines. The reports to relationship links an employee to their manager. The member of relationship links a person to a team or department. A knowledge graph is a network built from an ontology plus real data. Nodes in a knowledge graph represent entities such as people or places. Edges in a knowledge graph represent relationships defined by the ontology. A schema constrains what kinds of nodes and edges are allowed. RDF is a data format used to represent ontology based statements. An RDF triple consists of a subject, a predicate, and an object. OWL is a language used to define formal, machine readable ontologies. OWL supports reasoning, so software can infer new facts from existing ones. A reasoner checks an ontology for logical consistency. Inference in an ontology derives implicit facts from explicit ones. If a poodle is a dog and a dog is a mammal, a poodle is a mammal. This kind of inference relies on the transitivity of the is a relationship. Multiple inheritance occurs when a class has more than one parent class. A duck can inherit properties from both bird and swimmer classes. An ontology mapping links equivalent concepts across two different ontologies. Ontology alignment finds correspondences between classes in separate ontologies. Two ontologies can disagree even when they describe the same domain. A merged ontology combines concepts from multiple source ontologies. A cross domain linkage connects a medical ontology to a genetic ontology. Such a linkage lets a disease concept connect to the genes that cause it. A time ontology models concepts like instant, interval, and duration. The before relationship links two time intervals in sequence. The overlaps relationship links two time intervals that share some duration. An event ontology models actions, participants, and outcomes. The agent of relationship links an event to the entity that caused it. The patient of relationship links an event to the entity it affected. A causal relationship links one event to another it directly produces. A food ontology links ingredients, dishes, and cuisines together. The ingredient of relationship links a raw ingredient to the dishes that use it. The cuisine of relationship links a dish to its regional culinary tradition. A software ontology models classes, interfaces, and design patterns. The implements relationship links a class to an interface it fulfills. The extends relationship links a subclass to the class it inherits from. A depends on relationship links a software module to another it requires. A citation ontology models papers, authors, and the citations between them. The cites relationship links one paper to another paper it references. The authored by relationship links a paper to the people who wrote it. A concept map is a visual diagram of concepts and their linkages. Concept maps use labeled arrows to show the type of each relationship. Ontology engineering is the process of designing and maintaining an ontology. A competency question tests whether an ontology can answer a needed query. An ontology editor is software used to build and edit ontologies. Protege is a widely used tool for building OWL ontologies. Version control for ontologies tracks changes to classes and relationships over time. A linked data approach connects separate ontologies using shared identifiers. A shared identifier lets two systems agree they are talking about the same concept. Interoperability between systems improves when their ontologies share common linkages. A foundational ontology can serve as a bridge between unrelated domain ontologies. Bridging axioms declare that a concept in one ontology equals a concept in another. Without bridging axioms, two ontologies remain isolated even if their domains overlap.
